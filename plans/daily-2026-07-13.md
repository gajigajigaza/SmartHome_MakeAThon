## 1. weather.py 구현 — fetch_outdoor_weather

**수정/생성 파일**
- `dudeoji-api/weather.py` — 전면 구현 (기존엔 `NotImplementedError`만 던지는 뼈대)
  - `fetch_current_weather(lat, lon)` — OpenWeatherMap One Call API 3.0 호출, 기온/체감온도/습도/풍속/날씨상태/강수확률(`hourly[0].pop`) 반환
  - `fetch_air_pollution(lat, lon)` — Air Pollution API 호출, PM2.5/PM10/AQI 반환
  - `fetch_outdoor_weather(lat, lon)` — 위 둘을 `asyncio.gather`로 병렬 호출 후 병합 (async, `httpx.AsyncClient`, timeout 5초)
  - `_map_weather_condition(weather_id)` — OpenWeatherMap 코드(id)를 `recommendation_engine.py`가 실제로 비교하는 한글 문자열(`맑음`/`흐림`/`비`/`소나기`/`눈`/`태풍`)로 매핑
  - `_get_api_key()` — `os.getenv("OPENWEATHER_API_KEY")`, 없으면 `RuntimeError`
  - `fetch_outdoor_weather_mock(lat, lon)` — `dev_tools/mock_generator.py`와 같은 random 기반 패턴의 동기 목(mock) 함수, 테스트용
- `dudeoji-api/requirements.txt`, `dudeoji-api/pyproject.toml` — `httpx` 명시 추가 (환경엔 이미 설치돼 있었지만 선언 누락 상태였음)

**실행한 명령과 결과**
- `pip show httpx`, `pip show fastapi` → httpx 0.28.1 설치돼 있으나 fastapi의 직접 의존성은 아님(Required-by 없음) → 명시적으로 선언하기로 결정
- `python -c "import weather; ..."` → import 정상, `fetch_outdoor_weather_mock` 정상 동작 확인
- `PYTHONIOENCODING=utf-8 python -c "..."` (20회 반복) → `weather_condition` 값이 항상 `맑음/흐림/비/소나기/눈` 중 하나이고 `max_length=10` 제약을 만족함을 검증
- `_map_weather_condition` 샘플 id(200, 301, 500, 520, 531, 600, 701, 800, 801, 771, 781)에 대해 매핑 결과 육안 확인 → 의도대로 동작

**결정과 이유**
- 환경변수명은 `OPENWEATHER_API_KEY`로 통일 (사용자 지정, 기존 스켈레톤의 `WEATHER_API_KEY`는 폐기)
- `weather_condition` 매핑 함수가 핵심 — 실제 날씨를 가져와도 `recommendation_engine.py`가 기대하는 정확한 한글 문자열(`"비"`, `"소나기"`, `"눈"`, `"태풍"`)로 안 바꾸면 규칙 엔진이 인식 못 함. One Call 3.0의 `weather[0].id` 코드 그룹(2xx/3xx/5xx→비, 520-531→소나기, 6xx→눈, 771/781→태풍, 800→맑음, 나머지→흐림)으로 매핑
- 강수확률(`precipitation_probability`)은 One Call의 `current` 블록엔 없고 `hourly[0].pop`에만 있어서 그쪽에서 가져옴 (`exclude=minutely,daily,alerts`로 daily는 제외하되 hourly는 유지)
- mock 함수는 `dev_tools/`가 아니라 `weather.py` 안에 둠 — `dev_tools/`는 내 담당 파일 목록(`CLAUDE.md` 담당 범위)에 없어서 새 파일을 만들지 않기로 함
- `db.py`가 이미 `load_dotenv()`를 호출하므로 `weather.py`에 별도 dotenv 로딩 코드를 넣지 않음
- 실패(타임아웃/HTTP 에러) 시 try/except 없이 예외를 그대로 전파 (사용자 요청)

**새로 발견한 사실**
- `db.py`가 모듈 임포트 시점에 `load_dotenv()`를 호출하므로, `db`를 임포트하는 라우터 체인 안에서는 `.env`가 이미 로드되어 있음
- OpenWeatherMap Air Pollution API의 `pm2_5`는 µg/m³ 단위이고, `recommendation_engine.py`의 `is_dusty = pm25_value > 35.0` 임계값(한국 PM2.5 "나쁨" 등급 36 µg/m³~)과 단위가 일치함
- One Call API를 `units=metric`으로 호출하면 `wind_speed`가 m/s로 오고, `recommendation_engine.py`의 `is_windy = wind_speed_value > 8.0`과 단위가 일치함
- `savings.py`의 `estimate_savings(action)`은 현재 `action` 문자열 하나만 받고 pm25/wind_speed를 아예 쓰지 않음 — 즉 지금 시점엔 단위 불일치가 "발생할 수 없는" 상태 (TODO에 온도차/지속시간 반영 계획만 있고 pm25/wind_speed 반영 계획은 없음)
- `fetch_outdoor_weather()`가 저장소 어디에서도 호출되지 않음(grep으로 확인) — `readings_router.py`의 `POST /api/readings`는 클라이언트가 보낸 `weather_condition`/`pm25`/`wind_speed`를 그대로 쓸 뿐, `weather.py`와 연결돼 있지 않음
- `TEAM_STRUCTURE.md:132`가 "weather.py는 함수 자리만 있고 실제 호출 로직 없음"이라고 적혀 있는데, 이제 함수 본체는 구현됐지만 호출부 연결은 여전히 없어서 완전히는 스테일하지 않음 — 다만 갱신이 필요해 보임 (내 담당 파일 아님, 손 안 댐)

**미완 작업 / 이어갈 항목**
